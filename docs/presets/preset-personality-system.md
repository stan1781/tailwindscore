# Preset Personality System

TailwindScore preset is not a skin selector. It is a governed commerce design personality system.

## Purpose

Each preset defines a bounded commerce personality across shared SSR templates and the existing registry-governed runtime.

Preset personality exists to describe:

- visual identity
- commerce rhythm
- density profile
- motion personality
- mood family
- shell language
- content pacing

It does not exist to create:

- preset-specific templates
- preset-specific bundles
- arbitrary design controls
- visual builder branching

## Metadata Contract

Each preset must declare:

- `visual_identity`
- `commerce_rhythm`
- `density_profile`
- `motion_personality`
- `mood_family`
- `shell_language`
- `content_pacing`
- `capability_matrix`
- `governance_boundary`
- `localization_posture`

The metadata contract lives beside runtime preset definitions so governance language and runtime behavior do not drift apart.

## Capability Matrix

Each preset capability matrix must explicitly separate:

- supported capabilities
- unsupported capabilities

Supported capability examples:

- design tokens
- spacing rhythm
- motion rhythm
- density
- shell pacing
- content mood

Unsupported capability examples:

- template branching
- component structure changes
- runtime divergence
- arbitrary controls

If a requested behavior lands in the unsupported list, it is not a preset concern.

## Governance Boundary

Each preset must define a governance boundary that states:

- template branching: prohibited
- markup divergence: prohibited
- runtime feature fork: prohibited
- expansion policy: bounded

This boundary is not optional. If a preset requires alternate templates or logic branches, the request belongs to framework architecture, not preset expansion.

## Anti-Expansion Rules

- Presets may not become micro-themes.
- Presets may not own commerce logic.
- Presets may not introduce page-builder direction.
- Presets may not add independent frontend assets to simulate richer customization freedom.
- Presets may not redefine admin IA.

## Localization Posture

Each preset must describe:

- tone safety
- mood compatibility
- locale-aware fallback strategy

This keeps preset personality aligned with multilingual governance instead of assuming one tone works everywhere without degradation.
